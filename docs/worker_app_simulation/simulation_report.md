# 運用シミュレーション：入力動作における潜在的課題レポート

`WorkerApp.jsx`（作業員用日報アプリ）のコード定義・動作仕様をもとに、現場の運用において問題となりそうな入力動作やUXの抜け穴をシミュレーションしました。以下のリスクが想定されます。

## 1. 意図しないデータ消失のリスク

### 未送信での画面・日付遷移によるリセット
現状、時間が入力された状態で「作業日」や「現場を選択」を変更すると、`useEffect` がトリガーされて新しいデータをフェッチし、**未送信の入力中データが警告なしに上書き（消失）**してしまいます。
* **発生シナリオ:** A現場の時間を入力中に、「昨日のB現場の入力を確認しよう」と日付や現場を変更し、元の画面に戻ると入力が全て消えている。

### ブラウザ終了時のデータ消失（自動保存の欠如）
「下書き」機能は存在しますが、これは**「送信ボタンを押した際に通信エラーが発生した場合」**にのみ機能します。
* **発生シナリオ:** 作業時間を入力中、電話がかかってきてブラウザを閉じたり、アプリがメモリ不足でキルされた場合、入力中のデータは保存されずに消滅します。

## 2. 不正・不適切な時間の入力

### 同一現場内での時間の重複
異なる現場間の時間重複は `timeOverlapWarnings` によって警告されますが、**同一現場内の異なる作業項目間の重複はチェック対象から除外（`continue`）**されています。
* **発生シナリオ:** 同じ日の同じ現場で、「塗装」に 8:00〜17:00、「足場」にも 8:00〜17:00 と入力して送信できてしまい、総労働時間が16時間（2人工以上）として二重計上される恐れがあります。

### 逆転した時間・不整合な時間の入力
開始時間と終了時間に逆転した時間（例：開始 17:00、終了 08:00）を入力した際の、明確なUI上のブロックやエラーハンドリングが不足している可能性があります。
* **発生シナリオ:** 打刻ミスで負の時間が計算される、あるいは深夜業務として誤検知されて計算ロジックが破綻するリスクがあります。

## 3. データ品質の低下と管理コスト増大

### 一般作業員による「作業項目」の自由追加
`handleAddNewTask` 関数により、権限に関わらずどの作業員でも新しい作業項目を追加できます。
* **発生シナリオ:** 作業員ごとに「足場」「足場組立」「あしば」など表記揺れのある項目が乱立し、管理者側の原価計算や分析が困難になるリスクがあります。これは本来、管理者または職長のみが行うべき、もしくはマスターから選択すべき機能です。

### 過去日付の操作における「全体の進捗率」の誤操作
職長が「過去の日付（例：3日前）」を選択して日報を修正・送信した場合でも、送信時の処理で全体の `ProjectTasks` の進捗率パラメータが現在のスライダーの値で**現在の上書きとして**更新されてしまいます。
* **発生シナリオ:** 過去の日報の時間を少し修正しただけで、誤って全体の進捗率を過去の時点の数字（あるいは意図しない数値）に戻してしまう危険性があります。

---

## 💡 改善に向けた推奨アクション

1. **未保存警告の導入:** 現場や日付を変更する際、未送信のデータがある場合は `window.confirm` で警告を出す。
2. **同一現場内の重複チェック:** `timeOverlapWarnings` のロジックを修正し、同一現場内の作業であっても時間が被っている場合は警告を出す。
3. **作業項目の追加制限:** 作業項目の手動追加は「管理者」「職長」に限定するか、予め設定されたリストからの選択式に変更する。
4. **自動保存機能の強化:** 定期的に `localStorage` に入力中状態（フォームステート）を自動保存し、リロード後も復元できるようにする。
