# 🔧 フロントエンド改善提案書

> **対象**: 工事管理システム (cost-management-app)
> **分析日**: 2026-04-06
> **分析範囲**: AdminApp / WorkerApp / ScheduleViewApp + 全コンポーネント + ユーティリティ

---

## 📊 現状の総評

本システムは建設業向けの原価管理・工事進捗管理ツールとして、以下の主要機能を備えた実用的なアプリケーションです：

- **管理者用 (AdminApp)**: 8タブ構成のダッシュボード（ホーム/管理シート/実績入力/工事設定/作業員/配置表/システム設定/材料）
- **作業員用 (WorkerApp)**: モバイル向け日報入力インターフェース
- **閲覧用 (ScheduleViewApp)**: 読み取り専用の配置予定表

技術スタック: React 18 + Vite 5 + TailwindCSS 3 + Supabase + xlsx-js-style

**全体として正常に動作する堅牢なアプリケーションですが、コードベースの成長に伴い、以下の4つのカテゴリで改善の余地があります。**

## 1. 🏗️ アーキテクチャ改善

### 1-1. AdminApp.jsx の巨大化問題 ⭐ 優先度: 高

**現状の問題:**
[AdminApp.jsx](file:///c:/Users/katuy/Desktop/cost-management-app/src/AdminApp.jsx) は **1,041行・54KB** のモノリシックコンポーネントになっています。以下がすべて1ファイルに集約されています：
- 30以上の `useState`
- `fetchAllData` (約150行)
- Excel インポート/エクスポートのハンドラ
- CRUD 操作関数 (プロジェクト・タスク・レコード・作業員・協力業者)
- 全タブのレンダリングロジック

**改善案:**
| 分離先 | 内容 |
|---|---|
| `hooks/useProjects.js` | プロジェクト関連の state & CRUD |
| `hooks/useWorkers.js` | 作業員関連の state & CRUD |
| `hooks/useExcelImport.js` | Excel インポートロジック |
| `hooks/useAppSettings.js` | hourlyWage など全体設定 |

**メリット:** テスト容易性向上、可読性向上、バグ修正・機能追加時の影響範囲が限定される

**実装難易度:** 中（ロジックの移動のみ、UIへの影響なし）

## 2. 🎨 UX/UI 改善

### 2-1. ホーム画面にサマリー統計を追加 ⭐ 優先度: 高

**現状の問題:**
ホーム画面では工事一覧がカード形式で並んでいますが、**画面上部に全工事の合算値**（全体の予測粗利合計、施工中の現場数など）が表示されていません。個別カードの合計を頭の中で把握する必要があります。

**改善案:**
ホーム画面の最上部に「ダッシュボードサマリー」カードを追加。

```
┌───────────────┬────────────────┬──────────────┐
│ 施工中: 4件    │ 全体予測粗利     │ 総消化工数    │
│ 予定: 1件      │ ¥XXX,XXX       │ XXXh / XXXh  │
│ 完了: 0件      │                │              │
└───────────────┴────────────────┴──────────────┘
```

**実装難易度:** 低（`allProjectsSummary` は既に計算済み。表示を追加するのみ）

---

### 2-2. 実績入力タブのフィルタリング/検索 ⭐ 優先度: 高

**現状の問題:**
[InputTab.jsx](file:///c:/Users/katuy/Desktop/cost-management-app/src/components/tabs/InputTab.jsx) で全レコードがフィルタなしで表示されます。レコードが多い現場では、特定の日付・作業員で絞り込む手段がありません。

**改善案:**
- 日付フィルター（日付範囲指定 or 特定日のみ表示）
- 作業員フィルター
- フィルタの初期値を「今日」にする

**実装難易度:** 中

### 2-3. タブナビゲーションにアイコン追加 ⭐ 優先度: 中

**現状の問題:**
8つのタブが横並びになっており、タブ名がテキストのみで、素早い識別が少し難しいです。

**改善案:**
- 各タブにアイコンを追加（例: ホーム → 🏠, 管理シート → 📊, 実績入力 → ✏️ など）
- 小画面でのタブ表示をドロップダウンまたはアイコンのみに変更

**実装難易度:** 低

### 2-4. 削除操作のUndoサポート ⭐ 優先度: 中

**現状の問題:**
すべての削除操作が `window.confirm` → 即時削除 のフローになっています。誤操作で重要データを削除した場合、復元手段がありません。

**改善案:**
重要な削除操作（現場、作業項目）について：
- 即座にDBから削除せず、一定時間（5秒）の「元に戻す」トースト通知を表示
- トースト消滅後に実際のDB削除を実行

**実装難易度:** 中

---

### 2-5. 管理シートの進捗スライダーの改善 ⭐ 優先度: 低

**現状の問題:**
[DashboardTab.jsx](file:///c:/Users/katuy/Desktop/cost-management-app/src/components/tabs/DashboardTab.jsx) の進捗率スライダーの `step` が `5` になっているため、1%刻みの細かい進捗設定ができません。また、スライダーの `onMouseUp` でのみDB保存が走るため、キーボード操作では保存されません。

**改善案:**
- 進捗率の数値を直接テキスト入力可能にする
- `onChange` 後のデバウンス（300ms）でDB保存を発火

**実装難易度:** 低

### 2-6. モバイル対応の強化 ⭐ 優先度: 中

**現状の問題:**
AdminApp は基本的にPC向けのレイアウトです。テーブル系のUIは `overflow-x-auto` で対応していますが、工事設定や実績入力画面のフォーム要素がスマートフォンでは使いにくい可能性があります。

**改善案:**
- 管理者がモバイルで使う頻度が低ければ、現状維持で問題ありません
- 頻繁に使う場合は、テーブルの代わりにカード形式のリスト表示を検討

> [!NOTE]
> [WorkerApp.jsx](file:///c:/Users/katuy/Desktop/cost-management-app/src/WorkerApp.jsx) は既にモバイル最適化されており、良好な状態です。

## 3. ⚡ パフォーマンス改善

### 3-1. fetchAllData の過剰取得 ⭐ 優先度: 高（データ規模次第）

**現状の問題:**
[fetchAllData](file:///c:/Users/katuy/Desktop/cost-management-app/src/AdminApp.jsx#L47-L151) が起動時に **全プロジェクト・全タスク・全レコード・全協力業者レコード** を一括取得しています。プロジェクトや日報データが増加すると、初期ロード時間が大幅に増加します。

**改善案:**
- **第1段階 (推奨):** ホーム画面に必要な集計データのみを初回ロード。個別プロジェクトの詳細 (records, subcontractors) は該当タブを開いた時に遅延ロード
- **第2段階 (将来):** Supabase の RPC を使ったサーバーサイド集計

**実装難易度:** 高（データフローの大改修が必要）

> [!IMPORTANT]
> 現在のデータ規模が小さい場合（数十プロジェクト程度）は、この改善は急ぎではありません。データが増えてきた段階で検討してください。

---

### 3-2. 不要な再レンダリングの抑制 ⭐ 優先度: 中

**現状の問題:**
`AdminApp` コンポーネント内の `updateLayer` が呼ばれるたびに `setProjects` が更新され、全タブのコンポーネントが再レンダリングされます。

**改善案:**
- タブコンポーネントに `React.memo` を適用
- `summaryData`, `workerSummaryData` 等の `useMemo` は既に適切に使用されているが、子コンポーネントへの props として渡す関数に `useCallback` を適用

**実装難易度:** 低

---

### 3-3. 仕入帳データの初期ロード最適化 ⭐ 優先度: 低

**現状の問題:**
[PurchaseLedgerTab.jsx](file:///c:/Users/katuy/Desktop/cost-management-app/src/components/tabs/PurchaseLedgerTab.jsx) が独立して `fetchPurchaseData` を呼び出し、1000件ずつ全件をフェッチする while ループを使用しています。

**改善案:**
- サーバーサイドページネーション対応（現在はクライアント側ページネーション）
- ソート順を `id desc` にしてレコード数に依存しないページングに変更

**実装難易度:** 中

## 4. 🧹 コード品質改善

### 4-1. エラーハンドリングの統一 ⭐ 優先度: 高

**現状の問題:**
エラーハンドリングが統一されていません：
- `window.alert("...")` を使うケースと `alert("...")` を使うケースが混在
- エラーメッセージのフォーマットが不統一
- DB操作結果のエラーチェックが省略されているケースがある

```javascript
// handleSiteNameBlur: エラーチェックなし
const handleSiteNameBlur = async (id, newName) => {
    await supabase.from('Projects').update({ name: newName }).eq('id', id);
};
```

**改善案:**
- トースト通知コンポーネントを導入し、`window.alert` を廃止
- 共通のエラーハンドリングユーティリティを作成

**実装難易度:** 中

---

### 4-2. CSS のインラインクラスの整理 ⭐ 優先度: 低

**現状の問題:**
Tailwind のインラインクラスが非常に長くなっているケースがあります（特に配置表やホームのカード部分）。可読性に影響しています。

**改善案:**
- 頻出するスタイルパターンを `@apply` でカスタムクラス化
- または共通コンポーネント (Card, Button etc.) として抽出

**実装難易度:** 低

---

### 4-3. 未使用importの整理 ⭐ 優先度: 低

**現状の問題:**
[AdminApp.jsx](file:///c:/Users/katuy/Desktop/cost-management-app/src/AdminApp.jsx#L1-L2) で import されている lucide-react アイコンの中に、実際には直接使用されていないものがある可能性があります。

**改善案:**
使用されていない import を削除。ESLint の `no-unused-vars` ルールを導入。

**実装難易度:** 極低

---

### 4-4. 旧ファイル/一時ファイルの削除 ⭐ 優先度: 低

**現状の問題:**
プロジェクトルートに以下の不要と思われるファイルが残っています：
- `src/App.jsx.old` (47KB) — 旧メインアプリ
- `check-schema.mjs`, `check-schema2.mjs`, `check_data.js`, `check_delete.js` 等 — 開発時の一時スクリプト
- `test-xlsx.js`, `test_db.mjs`, `test_db2.mjs` — テストスクリプト
- `output.json`, `schema-output.json`, `tmp_err.txt` — 一時出力ファイル

**改善案:**
不要なファイルを削除し、リポジトリをクリーンに保つ。

**実装難易度:** 極低
