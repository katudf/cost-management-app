# 複数現場切り替え機能 実装計画

現在の実装では、現場のデータ（`siteName`, `masterData`, `records`, `progressData`）がフラットに管理され、それぞれが個別の `localStorage` キーに保存されています。
複数現場に対応するため、データを「現場（プロジェクト）」ごとにグループ化し、選択中の現場を切り替えられるように構造を変更します。

## データ構造の変更（マイグレーション）

現在は以下のような構造で保存されています：
- `cost-app-siteName`
- `cost-app-masterData`
- `cost-app-records`
- `cost-app-progressData`

これを以下のように、**現場の配列**と**現在選択中の現場ID**を持つ構造に変更します：
- `cost-app-projects`: 現場の配列。各要素に `id`, `siteName`, `masterData`, `records`, `progressData` を含める。
- `cost-app-activeProjectId`: 現在選択中の現場オブジェクトの `id`。

※初回ロード時に旧形式のデータが存在する場合は、新しい形式（一つの現場データ）としてマイグレーションする処理を組み込み、データロスを防ぎます。

## UI・操作の設計

### 追加するUI要素
- **ヘッダー部分**に「現在の現場」を表示するプルダウンメニュー（ `<select>` ）、または切り替え用モーダルを開くボタンを追加。
- メニュー内から「別の現場を選択」できるようにする。
- 「新しい現場を追加する」機能（ボタンと初期化処理）。
- プロジェクトを削除する機能（確認ダイアログ付き）。

### [MODIFY] `App.jsx`
1. **状態（State）の再構築**
   - `projects` と `activeProjectId` をベースとした状態管理に変更。
   - `activeProject` (現在選択中の現場データ一式) を導出する `useMemo` または単なる定数として定義。
2. **データの更新ロジック**
   - 既存の `updateMasterItem`, `addRecord`, `removeRecord`, ボリュームスライダーの更新処理などを、「`activeProjectId` に一致する `projects` の要素を更新する」処理へ書き換えます。
3. **新規現場のひな形**
   - 新規作成時、空白のレコードや進捗ゼロ状態のデフォルト `masterData` を持つプロジェクトオブジェクトを生成して `projects` に追加します。

## 検証プラン
1. 開発サーバー (`npm run dev`) で起動し、既存のデータ（前回テストで入力したもの）が「現場1」として正しく読み込まれる（マイグレーションされる）ことを確認。
2. 新しい現場を追加し、現場ごとに「日報実績」「進捗」などが完全に分離して保存・計算されることを確認。
3. ブラウザをリロードし、選択していた現場と、それぞれのデータが正しく復元されることを確認。
