# データベース連携運用への移行に向けた検討と計画案

## 背景と目的
現在、本アプリケーション（詳細工数管理システム）は、`Workers`（作業員リスト）のみをSupabaseから取得し、現場データ（プロジェクト、作業項目、入力実績、進捗）はブラウザの `localStorage` に保存する仕組みで動作しています。
今後、複数人での同時運用やデータの永続化を考慮し、システム全体をSupabase（PostgreSQL）の連携をベースとした運用に移行するための計画案をまとめます。

## 既存スキーマと現在のアプリとのマッピング分析
ご提示いただいた別プロジェクトのデータベーススキーマ（`Projects`, `ProjectTasks`, `DailyReports`, `Workers`など）は、現在のアプリケーションの機能に非常に親和性が高いです。

現在のアプリのデータ構造と、既存データベースのテーブルとの対応付け（マッピング案）は以下のようになります。

### 1. 現場（プロジェクト）データ
- **現状アプリ:** `projects` 配列内のオブジェクト (`id`, `siteName` 等)
- **データベース:** `Projects` テーブル
  - `name` カラムに現場名（`siteName`）を格納可能。
  - `status`, `startDate` などの追加情報も管理可能。

### 2. 工事設定（マスターデータ・目標設定）
- **現状アプリ:** 各プロジェクト内の `masterData` 配列 (`task`, `target`)
- **データベース:** `ProjectTasks` テーブル
  - `projectId` で現場に紐付け。
  - `status`（進行状態）、`order`（表示順）などを保持可能。
  - **※追加提案:** 現在のアプリが持つ「目標時間（target）」を保存するため、`ProjectTasks` テーブルに `target_hours`（数値型）等のカラムを追加する必要があります。

### 3. 実績入力（日報・工数記録）
- **現状アプリ:** プロジェクト内の `records` 配列 (`date`, `taskId`, `worker`, `hours`, `note`)
- **データベース:** `DailyReports` + `WorkLogs` + `Assignments` などが関連しますが、一番近いのは **`WorkLogs`** または **新しい実績専用テーブル** の作成です。
  - 既存の `WorkLogs` は `startTime` / `endTime` を持っていますが、現在のアプリは「`hours`（かかった時間）」として管理しています。
  - シンプルに連携させるなら、既存スキーマに合わせるか、実績用のテーブル（例: `TaskRecords`）を新規作成する方がスムーズかもしれません。

### 4. 進捗データ
- **現状アプリ:** プロジェクト内の `progressData` オブジェクト（タスクIDごとの％）
- **データベース:** 
  - `ProjectTasks` テーブルに `progress_percentage`（進捗率・数値型）カラムを追加して保存するのが最適です。

## 移行に向けた推奨アプローチ（フェーズ提案）

一気にすべてをDB対応させるとシステムが動かなくなるリスクがあるため、段階的（フェーズごと）な実装を提案します。

### フェーズ1: データベース（テーブル/カラム）の準備
1. **既存テーブルの活用:** `Projects` と `ProjectTasks` を再利用する。
2. **必要なカラムの追加:**
   - `ProjectTasks` テーブルに `target_hours`（目標時間）と `progress_percentage`（進捗率）を追加。
3. **新規テーブルの作成（実績管理用）:**
   - 既存の複雑な日報構造（`DailyReports` / `WorkLogs`）を使うより、現在のアプリの入力フォーマットに合わせたシンプルな実績テーブル `TaskRecords` を新設することをおすすめします。
     - カラム案: `id`, `project_id`, `project_task_id`, `worker_name` (または `worker_id`), `date`, `hours`, `note`

### フェーズ2: 新規プロジェクト作成・ExcelインポートのDB化
- Excelからインポートした際に、`localStorage` ではなく Supabase の `Projects` テーブルに新しい現場を、`ProjectTasks` テーブルに作業項目を一括で `INSERT` するよう処理を書き換えます。

### フェーズ3: 実績入力と進捗のDB化
- 日報（実績入力）を登録・削除する際、Supabase の実績テーブルに対して操作を行うように変更。
- 進捗バーを動かした際、`ProjectTasks` テーブルの進捗率を `UPDATE` するように変更。
